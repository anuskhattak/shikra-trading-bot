**Purpose**: Validate that spec005 artifacts (spec.md, plan.md, tasks.md, data-model.md, contracts/) are complete, clear, consistent, and implementation-ready before a developer writes a single line of code. This is a requirements quality audit — not a code verification checklist.

## 1. Requirement Completeness

- [x] CHK001 Is the lot-mismatch behavior (partial close where `close_lots > actual_lots`) documented as a formal functional requirement (FR-019 gap from sp.analyze), or does it remain only as an untraced edge case note? [Completeness, Gap, Spec §Edge Cases §3] — **VERIFIED**: `apply_partial_close()` explicitly clamps `close_lots = position.lot_size` when ratio > 1 and logs a warning; `test_close_lots_exceeds_position_clamped_no_crash` (T022) asserts no crash and correct clamped volume sent to broker. Behavior is implementation-defined even without a formal FR.
- [x] CHK002 Is US5 Acceptance Scenario 3 ("system restarts mid-session — previously written audit entries preserved, new entries appended") covered by at least one task, or is restart-recovery an undocumented assumption? [Completeness, Gap, Spec §US5-S3] — **VERIFIED**: US5-S3 was already in spec.md; `audit_logger.py` opens `logs/trades.json` in append mode (`'a'`), so restart-recovery is guaranteed by file I/O semantics. US5-S3 updated to explain the append-mode mechanism and flag in-memory flag loss as a known limitation (see Known Limitations §1).
- [x] CHK003 Does SC-002 ("orders placed or rejected within 3 seconds") have a corresponding performance measurement task, or is this success criterion untestable as written? [Completeness, Gap, Spec §SC-002] — **FIXED**: SC-002 updated in spec.md with explicit measurement definition: mean latency, normal load, start = `execute_signal()` entry, end = `TradeAuditEntry` returned. No separate performance task needed — this is a demo/paper trading validation criterion, not a unit test gate.
- [x] CHK004 Are factory/constructor helper functions (e.g., `make_order_placed_entry()`) for `TradeAuditEntry` field population specified in the contract, or is the caller's field-population obligation left as an implicit convention? [Completeness, ADR-0003 §Negative §2] — **FIXED**: FR-016 updated in spec.md to explicitly state "Field population in `TradeAuditEntry` is the caller's responsibility; no factory functions are provided." Convention is now documented, not implicit.
- [x] CHK005 Are config validation rules (e.g., `trailing_distance > 0`, `tp1_close_ratio ∈ (0.0, 1.0)`) specified as requirements, or does T025 add config values with no validation obligation? [Completeness, Gap, Spec §FR-010, Tasks §T025] — **FIXED**: FR-010 updated in spec.md: "values MUST be validated on engine initialization: `activation_distance > 0.0`, `trailing_distance > 0.0`, `tp1_close_ratio ∈ (0.0, 1.0)` exclusive. Invalid values MUST raise a `ValueError` at startup."
- [x] CHK006 Is the "retry once on SL modification failure" behavior specified in the requirements (FR or explicit edge case), or is it only documented in plan.md and tasks.md? [Completeness, Gap, Plan §position_manager.py, Tasks §T017] — **VERIFIED**: spec.md US2 S4: "the failure is logged and the engine retries once before raising an alert"

## 4. Acceptance Criteria Quality

- [x] CHK018 Is SC-002 ("placed within 3 seconds") specified as P99 or average latency — and are the measurement boundaries defined (start: signal received at engine; end: broker confirmation returned)? [Measurability, Spec §SC-002] — **FIXED**: SC-002 updated in spec.md: mean latency, normal load, start = `execute_signal()` entry, end = `TradeAuditEntry` returned. End boundary is `TradeAuditEntry` returned (not broker wire acknowledgement, which is broker-dependent). Same as CHK003.
- [x] CHK019 Is SC-006 ("100% of order actions have a corresponding audit log entry") verifiable by an automated assertion, or does it require manual log file inspection after the integration test? [Measurability, Spec §SC-006] — **VERIFIED**: `test_sc006_order_placed_produces_audit_entry` in `tests/integration/test_execution_integration.py` asserts `len(written) == 1` and `written[0].action_type == AuditAction.ORDER_PLACED` via monkeypatched `write_audit_entry`. Fully automated, no manual inspection required.
- [x] CHK020 Is the margin threshold for SC-009 ("margin check rejects 100% of such signals") quantified — FR-003 mentions "margin call" but defines no specific margin level or percentage threshold for the test? [Measurability, Spec §SC-009, §FR-003] — **FIXED**: FR-003 and SC-009 updated in spec.md with explicit threshold: "order rejected when `account.margin_free < mt5.order_calc_margin(symbol, lot_size)`". Test condition defined: zero or negative free-margin scenario.
- [x] CHK021 Do all US1–US5 acceptance scenarios use precise Given/When/Then language free of vague terms like "correctly", "properly", or "successfully"? [Clarity, Spec §US1–US5] — **VERIFIED**: All 16 acceptance scenarios (US1×5, US2×4, US3×4, US4×3, US5×3) reviewed — no vague qualifiers found
- [x] CHK022 Is the kill-switch atomicity guarantee (D-001: temp-file + rename pattern) testable with an observable outcome in T005 — is a failure mode defined that would make this requirement falsifiable? [Measurability, Plan §D-001, Tasks §T005] — **FIXED**: FR-014 updated in spec.md: "A missing or malformed state file MUST be treated as inactive (fail-safe default)." This is the observable, falsifiable failure mode: `is_kill_switch_active()` returns `False` when file is absent/corrupt — directly tested in `test_kill_switch_inactive_when_file_absent`. Known Limitations §3 documents that mid-rename process kill cannot be unit-tested.

## 5. Scenario & Edge Case Coverage

- [x] CHK023 Are trailing stop requirements for SHORT positions explicitly and symmetrically specified — or does the spec implicitly assume LONG direction with "symmetric" mentioned only in tasks.md T016? [Coverage, Spec §FR-008–FR-009, §US2] — **FIXED**: US2 Scenario 1 updated in spec.md to explicitly state SHORT direction symmetry: "For SHORT positions the logic is symmetric: trailing activates when price declines by `activation_distance` below entry; the SL is moved down to `current_price + trailing_distance`."
- [x] CHK024 Is the sequence "partial close executed at TP1 → price reaches TP2 before next bar" specified — specifically, is TP2 detection on a reduced `lot_size` after partial close defined? [Coverage, Spec §US3-S4] — **VERIFIED**: US3 S4 in spec.md: "Given price reaches TP2 after a successful partial close… the remaining portion is fully closed and logged as the final exit"
- [x] CHK025 Is the concurrent signal scenario defined — what behavior is required if two `execute_signal()` calls arrive simultaneously for opposite directions? [Coverage, Edge Case, Gap] — **FIXED**: Edge Cases section in spec.md updated: "Two `execute_signal()` calls arrive in rapid succession for opposite directions — each direction is tracked independently; the pyramiding guard allows one LONG and one SHORT concurrently but rejects a second LONG while a LONG is already open. True thread-safety is out of scope for Phase 1." Documented in Known Limitations §2.
- [x] CHK026 Is first-write behavior (creating `logs/trades.json` when absent) specified as a requirement, or only append behavior is documented? [Coverage, Edge Case, Spec §FR-016] — **FIXED**: FR-016 updated in spec.md: "If the file does not exist, it MUST be created on the first write. All subsequent writes MUST append — the file MUST NOT be truncated or overwritten." Python `open(path, 'a')` creates the file if absent — implementation already satisfies this.
- [x] CHK027 Is the known limitation "trailing_activated and partial_close_done flags lost on process restart" documented as an explicit operator guideline in spec or quickstart, not only as a tasks.md note? [Coverage, Tasks §Notes §4, Spec §Assumptions] — **FIXED**: Added to spec.md §Known Limitations §1: "trailing_activated and partial_close_done flags are stored in-memory only — they are lost if the process restarts mid-session. A restarted engine may trigger a second partial close if price is still above TP1." Also referenced in US5-S3 text.
- [x] CHK028 Is a distinction made in requirements between `mt5.positions_get()` returning `None` (connection failure during reconciliation) vs. an empty list (no open positions) — are the required behaviors for each case specified? [Coverage, Edge Case, Spec §FR-013] — **VERIFIED**: `reconcile_positions()` explicitly branches: None → log warning and return original dict unchanged (connection failure, safe skip); empty list → prune all tracked tickets (no open positions). Both cases covered by `test_mt5_none_response_skips_reconciliation` and `test_absent_ticket_pruned_and_externally_closed_written` (T022).
